## What Went Wrong

### 1. PR creation failed — token permissions
`gh pr create` exited with `Resource not accessible by personal access token (createPullRequest)`. This is a recurring issue in this repo (seen in prior sessions). The agent had no fallback beyond reporting the URL — which was sufficient but not ideal.

### 2. gstack upgrade signal (UPGRADE_AVAILABLE 1.6.1.0 → 1.14.0.0) was never handled
The /ship preamble before compaction printed `UPGRADE_AVAILABLE`. The gstack skill requires reading `gstack-upgrade/SKILL.md` and following the inline upgrade flow. After compaction, this signal was lost and never surfaced to the user. The upgrade remains pending.

### 3. HAS_ROUTING: no / ROUTING_DECLINED: false prompt was never handled
Same cause — the /ship preamble signal was lost at compaction. The routing rules injection into CLAUDE.md was never offered to the user.

### 4. The /ship workflow was abandoned in favor of a direct git-agent invocation
The user invoked `/ship` twice (before compaction), then after compaction did not re-invoke it. The git-agent was used directly instead. This bypassed ship's Step 1 pre-flight, Step 3 review, and the gstack-managed flow. The result was correct but the workflow divergence went unacknowledged.

## Root Cause Analysis

| Issue | Root Cause | Impact |
|-------|------------|--------|
| PR token failure | GitHub PAT lacks `pull_requests: write` scope — known env constraint | Low — user can create PR manually via URL |
| gstack signals lost | Compaction occurred mid-/ship preamble; no artifact written before context collapsed | Medium — upgrade pending, routing rules not injected |
| /ship abandoned | User shifted to git-agent directly after compaction; agent did not re-surface the incomplete /ship flow | Low — git ops completed correctly |
