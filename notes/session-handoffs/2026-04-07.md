## What Was Done

### Bug Fix: False "Unsaved Changes" Dialog on Recipe/Dish Navigation

**Problem:** Opening any recipe or dish in the recipe-builder and immediately navigating away (without touching anything) triggered the "unsaved changes" confirmation modal.

**Root cause investigation** used `/browse` live browser testing + console logging injected into `hasRealChanges()` to expose the exact snapshot diff. Three separate post-init form mutations were found:

| # | Snapshot field | Post-init trigger | Mechanism |
|---|---|---|---|
| 1 | `ingredients[].total_cost` | `refreshAllLineCalculations()` in `RecipeIngredientsTableComponent` | `effect()` watches `products_()`/`recipes_()` signals → fires after state loads → `setValue(lineCost)` on each ingredient |
| 2 | `total_weight_g` (form level) | Same effect chain | `setValue(total_cost)` propagates → `ingredientsArray.valueChanges` → `updateTotalWeightG()` → new computed weight written to form |
| 3 | `yield_conversions[0].amount` | `RecipeHeaderComponent.effect()` auto-sync | `computedYieldAmount_()` reads `totalWeightG` → rounds to 2dp (`Math.round(x*100)/100`) → `syncYieldFromMetrics()` → `setValue(0.45)` overwrites stored `0.4465` |

**Fix** — all in `recipe-builder.page.ts` → `getRecipeSnapshotForComparison()`:
- Removed `total_cost` (per-ingredient) from `ingNorm`
- Removed `total_weight_g` and form-level `total_cost` from `normalized`
- Changed `yieldNorm` amount from `Number(c?.amount)` → `Math.round(Number(c?.amount) * 100) / 100`

**Verified:** `/browse` confirmed zero dialog on navigation after full state load (5s wait).
