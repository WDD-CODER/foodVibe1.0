## Section 8: Plain-Language Workflow Narrative

**Session start — automated gate chain.** When the developer opens Claude Code in the FoodVibe 1.0 project, two things happen automatically before any interaction: the `scripts/session-startup.sh` SessionStart hook fires, loading the most recently modified `docs/session-state-*.md` file for the current branch and injecting a "SAVE TARGET" line that tells Claude where to write session state at close. Then CLAUDE.md loads as project context, mandating that Claude read `agent.md` and `.claude/copilot-instructions.md` before doing any work. Once Claude confirms "Yes chef!", the session is live and Claude has the full rule set, agent roster, skill triggers, and MemPalace search mandate loaded.

**Day-to-day feature work — plan-execute loop.** Feature work follows a strict flow: `/new-feature` for scoping, `/brief` to crystallize requirements into `sessions/*/brief.md`, `/plan-implementation` to verify the plan against live code (flags gaps, pauses for approval), then `/execute-it` to run it. After execution, `/save-plan` writes plan files to `plans/` and syncs subtasks to `.claude/todo.md`. This is the most-traveled path in the workflow and is fully functional. The branch-guard PreToolUse hook fires automatically before every file edit, ensuring Claude never commits code to `main`.

**Background automation — what actually runs vs. what's documented.** Three automated systems run without user intervention: (1) The git post-commit/post-merge hooks invoke `embed-runner.js` to index changed files into MemPalace after every commit — this runs silently and is the source of the semantic search capability. (2) The context-monitor PostToolUse hook fires after every tool call, tracking transcript size and alerting Claude at 40%/60%/70% thresholds with a hard stop at 70% forcing a session handoff. (3) An Anthropic Cloud Routine fires nightly to run `/nightly-audit`, scanning for 6 violation categories. The reflect automation — where `auto-reflect.ps1` was supposed to trigger `/reflect AUTO` at session close — does NOT run. It has never been wired into `settings.json`. All reflect runs are manual.

**Session end — 14-phase pipeline.** When the developer signals session end (either via `/session-handoff`, `/end-session`, or the session close), the end-of-session-agent runs a 14-phase checklist: build verification, techdebt scan (writes a new rolling report to `techdebt-reports/`), breadcrumb/doc update, git commit and push via git-agent, and writing the session handoff to both `.claude/sessions/` and `docs/session-state.md`. The Stop hook (`handoff-check.sh`) validates that `docs/session-state.md` contains a commit hash, summary, and next steps before the session is considered closed. The intended connection from the Stop hook to `auto-reflect.ps1` does not exist.

**What the workflow does well vs. where it has gaps.** The workflow excels at session continuity (state is reliably preserved and loaded across sessions), code quality enforcement (branch guard, no-semicolons pre-commit, context monitor), and structured feature delivery (plan→execute flow with validation gates). The gaps are concentrated in the reflect automation leg (D-19 is the most important open issue), the embedding pipeline (unverified end-to-end), and `settings.json` hardening (single point of failure for all hook automation with no recovery runbook).
